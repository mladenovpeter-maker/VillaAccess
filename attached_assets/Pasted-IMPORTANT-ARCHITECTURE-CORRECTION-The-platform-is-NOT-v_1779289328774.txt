IMPORTANT ARCHITECTURE CORRECTION

The platform is NOT villa-specific at the hardware layer.

Cameras, gates, barriers, and intercoms are SHARED infrastructure for the entire property/community.

Do NOT directly bind:
- cameras to villas
- gates to villas
- intercoms to villas

Instead:

====================================================
CORRECT REAL-WORLD ARCHITECTURE
====================================================

There are:

- multiple villas
- multiple shared entrances/exits
- multiple shared ANPR cameras
- multiple shared intercoms
- shared gates/barriers

Example:

Entrance A:
- Camera A1
- Camera A2
- Intercom A

Entrance B:
- Camera B1
- Camera B2
- Intercom B

These entrances are shared by ALL villas.

The access decision is based on:
- reservation validity
- assigned vehicle
- PIN validity
- access permissions

NOT villa-specific camera ownership.

====================================================
NEW PLATFORM LOGIC
====================================================

The system should support:

1. Shared Entrances
Entrances should be configurable objects.

Each entrance may contain:
- multiple cameras
- multiple intercoms
- gate/barrier controls

2. Villas
Villas are logical reservation/access units only.

Villas do NOT own hardware devices.

3. Reservations
Reservations grant temporary access rights.

A reservation contains:
- guest name
- villa assignment
- check-in/check-out dates
- assigned vehicle(s)
- PIN code
- access validity

4. Access Validation
When a vehicle arrives:

System checks:
- reservation validity
- assigned vehicle
- active access window
- confidence score

If valid:
- open shared gate/barrier
